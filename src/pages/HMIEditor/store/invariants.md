1. Если `parentId === null`, то `node.id` должен быть в `pages[activePageId].rootIds`
2. Если `parentId === P` то `node.id` должен быть в `nodes[P].childrenIds`
3. `childrenIds` остаётся источником порядка (parentId порядок не заменяет)
4. Любая операция “переподвесить/вставить/вырезать/группировать” должна обновлять обе стороны
5. Любая операция, которая влияет на структуру должна возвращать в патче `tree: true`
6. если после удаления у группы childrenIds пуст, удаляем группу тоже и повторяем вверх по parentId

## 1) Инварианты архитектуры стора

- **Все изменения “проектных” данных проходят через команды** (commands), единая точка входа — `runCommand(...)`.
- **Домены/утилиты — чистые**: получают входные данные → возвращают результат;
  **не знают про store**, не трогают UI/selection/dirty напрямую.
- **Никаких “сервисов-оркестраторов, которые знают обо всём”**: оркестрация живёт в командах; домены — детали.
- **Политики (dirty, selection, undoable, reindex) задаются декларативно из команды** (не размазаны по куче if-ов в середине логики).

---

## 2) Контракт `runCommand` и патчей

- Команда либо **ничего не делает** (`return null`), либо возвращает **детерминированный результат**.
- **Патчи плоские** (как ты и говорил): команда не должна “полуперезаписывать” вложенные структуры не своей ответственности. Для `bindings/events/...` — отдельные команды/домены.
- **COW-поведение обязательно**: если сущность не менялась — её ссылка остаётся прежней.
  Это критично для perf при `useNodeStore(s => s.nodes[id])`.
- Команда не должна массово пересоздавать большие объекты без нужды (например, `nodes`, `pages`, `varIndex` целиком) — только при реальных изменениях.

---

## 3) Инварианты модели дерева узлов (nodes/pages)

- `pages[activePageId]` существует, иначе команда **ничего не делает** (page guard).
- `page.rootIds` содержит **только существующие** nodeId.
- Для каждого `nodeId` из `rootIds` и `childrenIds`:
    - `nodes[nodeId]` существует,
    - `nodes[nodeId].id === nodeId`.

- **Нет циклов** в дереве (узел не может оказаться предком самого себя).
- Если у узла есть `parentId`, то:
    - этот parent существует,
    - узел присутствует в `parent.childrenIds`,
    - и наоборот: если `id` в `childrenIds`, то у ребёнка `parentId` указывает на этого родителя.

- Каждый узел принадлежит **ровно одному контейнеру**: либо в `rootIds`, либо в `childrenIds` ровно одного родителя (не “в двух местах”).

---

## 4) Инварианты порядка слоёв (z-order)

- Порядок отрисовки = порядок в массиве контейнера:
    - на корне это `page.rootIds`,
    - внутри группы это `group.childrenIds`.

- Команды типа bringToFront/moveUp… работают **только через** единую функцию перестановки (`reorderRootIds`/аналог), чтобы не плодить несогласованные правила.
- Группировка заменяет детей на группу **в предсказуемой позиции** (ваш `replaceChildrenWithGroup` с pivot min/max):
  группа появляется в контейнере там, где договорились (“min” или “max” индекс детей).

---

## 5) Инварианты групп и аффинных преобразований

- При группировке/разгруппировке соблюдается правило:
    - **визуальное положение на сцене не меняется**, меняются только системы координат.

- Дети группы хранятся в **локальных координатах группы** (после `makeChildrenLocal`).
- При разгруппировке дети становятся **глобальными** (через обратное преобразование).
- После любого изменения состава/трансформаций в группе:
    - либо пересчитывается bbox/матрицы группы,
    - либо гарантируется корректность через `recalcGroupsUpwardsCOW` (вверх по цепочке).

- Пустые группы либо запрещены, либо **каскадно удаляются** по правилу (и это правило одно, централизованное), чтобы не было “висячих” контейнеров.

---

## 6) Инварианты selection (выделение)

- Изменение selection — **политика команды**, а не побочный эффект домена/утилит/бывших сервисов.
- У команды есть явное решение:
  **keep / clear / set([...])** (или эквивалентный протокол).
- После операций:
    - **group** → selection переходит на новую группу,
    - **ungroup** → selection переходит на бывших детей (или на корневой набор — главное, чтобы правило было единым),
    - **remove** → selection очищается, если удаляли выделенные (ваша мысль про `resetSelection=true`).

---

## 7) Инварианты dirty (изменённость проекта)

- `dirty`/`meta.isDirty` меняется **только** на “смысловых” командах, которые меняют проектные данные.
- Транзиентные вещи (scrub/drag/preview/konva direct) **не обязаны** попадать в dirty и тем более в undo.
- `open/close/new` задают `dirty=false` в одном месте, без “протекания” dirty в остальные домены.

---

## 8) Инварианты индексов (bindings index)

Из нашего обсуждения про “смешение доменов” и твоё “мне больше нравится индексация в командах”:

- `varIndex`/`nodeIndex` — **derivative state**, он должен быть **строго консистентен** с `nodes`.
- Любая команда, которая:
    - добавляет/удаляет узлы,
    - меняет bindings узла,
    - клонирует/вставляет поддеревья,
      обязана **обновить индексы в рамках той же команды** (атомарно).

- Правило “1 привязка на параметр” (как мы фиксировали):
  для `(nodeId, property)` должна существовать максимум одна binding-запись.
- Binding-структуры не должны тащить лишнюю идентификацию, если она не нужна (ты отмечал избыточность `id` у Binding) — т.е. стабильность достигается через `(nodeId, property)`.

---

## 9) Инварианты clipboard (copy/cut/paste)

- Clipboard хранит **поддеревья** (узел + все потомки) и достаточные данные для восстановления.
- Paste всегда делает:
    - `rehydrate` id (новые id, корректная карта old→new),
    - чинит `parentId/childrenIds`,
    - вставляет rootIds в контейнер с соблюдением z-order,
    - обновляет индексы,
    - выставляет selection на вставленные узлы (если политика такая).

- Cut = Copy + Remove, но важно: Remove должен быть тем же “правильным” (с каскадами групп/индексами/selection).

---

## 10) Инварианты производительности и частых апдейтов

- При `useNodeStore(s => s.nodes[id])` и размере `nodes` до ~5000:
    - изменяем **только** затронутые узлы (и минимальные контейнеры),
    - не пересоздаём весь `nodes`/`pages`/индексы без необходимости.

- Частые RAF-апдейты (drag/scrub):
    - либо идут мимо стора (как ты сейчас делаешь напрямую в Konva),
    - либо идут в отдельный “не-undoable/не-dirty” канал,
    - а **коммит** делается на завершение жеста (pointerup/enter/blur/unmount).

- Ввод чисел: если значение менялось локально и не было commit — **на размонтировании** делаем финальный commit (твоя идея с `useEffect cleanup` + `lastNumberRef`).

---

## 11) Инварианты валидации/нормализации проекта при загрузке

- AJV валидирует схему, но поверх этого есть **санитайзер/нормализатор**:
    - чинит/дедупит массивы id,
    - гарантирует наличие `parentId`,
    - удаляет/репортит “битые ссылки”,
    - в strict-режиме падает, в repair-режиме чинит и собирает issues.

- После `open(project)` состояние должно удовлетворять всем инвариантам из разделов 3–9.

---

## 12) Инвариант “простота команд” (Sonar cognitive complexity)

- Любая команда держится в пределах лимита (у тебя 15).
- Если команда начинает разрастаться:
    - выносим расчёты в домены,
    - политики — в компактный протокол результата (`selection`, `dirty`, `undoable`, `reindex`),
    - вместо вложенных if-ов — ранние выходы и мелкие функции.
